# bin_analyzer

Split-pane binary analyzer that maps ELF debug info bidirectionally between source code and disassembly.

---

## Building

```
cd tools/bin_analyzer
zig build
```

Binary is at `tools/bin_analyzer/zig-out/bin/bin_analyzer`.

---

## CLI Mode

Non-interactive queries that dump results to stdout. Useful for scripting or when a terminal is unavailable.

```
bin_analyzer <elf-binary> [options]
```

### Options

| Flag | Description |
|------|-------------|
| `--list-files` | List all source files present in the binary's DWARF debug info. |
| `--source <file:line>` | Show source context around a line and its corresponding disassembly. `file` is a substring match against debug info paths. |
| `--disasm <addr>` | Show disassembly context around an address (hex, with or without `0x` prefix) and the corresponding source location. |
| `--func <name>` | Find a function by name (substring match against objdump labels), show its disassembly and source location. |
| `-C, --context <n>` | Number of context lines around matches (default 5). |

### Examples

```
# List all source files compiled into the kernel
bin_analyzer kernel.elf --list-files

# Show source and disassembly for a specific line
bin_analyzer kernel.elf --source "kernel/main.zig:50"

# Look up a function by name
bin_analyzer kernel.elf --func kMain

# Show disassembly at an address with 10 lines of context
bin_analyzer kernel.elf --disasm 0xffffffff800330d0 -C 10
```

## TUI Mode

Launch with no flags for the interactive split-pane viewer:

```
bin_analyzer <elf-binary>
```

Left pane shows source code, right pane shows disassembly (Intel syntax). Moving in one pane highlights and auto-scrolls the corresponding lines in the other.

### Keybindings

| Key | Action |
|-----|--------|
| `j` / `k` | Move cursor down / up |
| `h` / `l` | Move character cursor left / right within the line |
| `w` / `b` | Jump to next / previous word |
| `0` / `$` | Jump to start / end of line |
| `Tab` | Switch active pane (source / disasm) |
| `Ctrl-D` / `Ctrl-U` | Half-page down / up |
| `/` | Open file search prompt (substring match against debug info paths) |
| `gd` | Go to definition (see below) |
| `gb` | Go back (pop navigation stack) |
| `q` / `Ctrl-C` | Quit |

### Go to Definition (`gd`)

- **In disasm pane:** If the cursor is on a `call` or `jmp` instruction, jumps to the target address. The source pane updates to show the target function's source.
- **In source pane:** Extracts the identifier under the cursor and searches disasm labels for a matching function name. Jumps to the first instruction of that function.

Each `gd` pushes the current position onto a navigation stack. `gb` pops and restores the previous position.

### File Search (`/`)

Type a substring to match against all source file paths in the binary's debug info. On Enter, loads that file in the source pane and jumps the disasm pane to the first instruction attributed to that file. Escape cancels.

## How It Works

### Data Sources

1. **DWARF debug info** -- Parsed from the ELF binary using `std.debug.Dwarf`. Provides bidirectional mappings between PC addresses and source locations (file, line, column).
2. **Disassembly** -- Generated by shelling out to `objdump -d -M intel`. The output is parsed to extract instruction addresses, mnemonics, and operands. Hex byte dumps and continuation lines are stripped.

### Bidirectional Mapping

At startup, the tool builds two mapping directions:

- **PC to source** (forward): Uses `Dwarf.getLineNumberInfo(address)` to resolve any instruction address to a file and line number.
- **Source to PC** (reverse): Iterates every compile unit's `.debug_line` table via `Dwarf.populateSrcLocCache()`, collecting all PC addresses for each `(file, line)` pair. These PCs are then mapped to disasm line indices for direct lookup.

The forward map is used when navigating in the disasm pane (cursor moves, highlight the corresponding source line). The reverse map is used when navigating in the source pane (cursor moves, highlight the corresponding disasm instructions).

### ELF / DWARF Loading

The ELF file is read into memory and section headers are scanned for `.debug_*` sections. Each section's data is registered with a `std.debug.Dwarf` struct, then `dwarf.open()` is called to parse abbreviation tables and compile units. This follows the same pattern as `kernel/utils/elf.zig`.

### TUI Rendering

Raw terminal mode via POSIX termios (disable canonical mode, echo, signals). ANSI escape codes for cursor positioning, colors, and the alternate screen buffer. All output is batched into a buffer and flushed in a single `writeAll` to avoid flicker.
